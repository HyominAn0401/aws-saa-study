### 📝 666. 재해복구 (DR)
Oracle 데이터베이스를 최신 버전으로 업그레이드, DR 설정 및 일반 운영에 대한 운영 오버헤드 최소화. DB의 기본 운영 체제에 대한 액세스 유지 필요 

---

### 🎯 1. 요구사항 분석
* **재해 복구(DR)설정**: 다른 가용 영역보다 다른 리전에 필요
* **DR 설정 운영 오버헤드 최소화**: 완전 관리형 서비스 사용 필요
* **DB 기본 운영 체제에 대한 액세스 유지**: 기존 완전 관리형 서비스에 Custom 필요 

---

### 🔍 2. 핵심 서비스 분석 

| 서비스/용어명 | 핵심 특징 | 비고 |
| :--- | :--- | :--- |
| **교차 리전 자동 백업** | 한 리전의 DB 백업을 다른 리전으로 자동으로 복제해두는 기능<br>데이터 손실을 최소화하려는 요구사항<br>실시간 복제본이 아니라 실제 DB로 돌리려면 해당 백업으로 복원하는 과정 거쳐야 함 | 백업 데이터 전송 비용, 리전 간 저장 비용 발생 | 
| **Amazon RDS Custom**| 보통 RDS는 AWS가 OS 관리부터 다 해주지만 사용자가 OS에 접근할 수 없음. Custom은 사용자가 OS에 SSH 접속이 가능<br>특정 레거시 애플리케이션이 DB의 OS 설정을 건드려야 할 때<br>DB엔진에 특수한 패치나 타사 소프트웨어를 설치해야 할 때 | Oracle, Microsoft SQL Server에서 주로 쓰임 |
| **가용 영역** | 같은 도시 내<br>동기식 복제(메인 DB에 쓰면 옆 AZ에도 써짐) | ***고가용성*** |
| **리전** | 국가/대륙 간<br>비동기식 복제 | ***재해 복구*** |

---

### ✅ 3. 선택지 분석 및 풀이

- [ ] **A**: EC2 인스턴스로 마이그레이션 + 다른 리전으로 DB 복제
    - **이유**: 운영 오버헤드가 큼 
- [ ] **B**: Oracle용 RDS로 마이그레이션 + 교차 리전 자동 백업 활성화 
    - **이유**: 
        - OS에 대한 액세스 유지 조건 미충족
        - 백업본 복제는 '복원' 과정이 필요하므로 실시간성 DR 측면에서 읽기 복제본보다 복구 목표(RTO)가 김
- [X] **C**: Oracle용 RDS Custom으로 마이그레이션 + 다른 리전 DB에 대한 읽기 전용 복제본 생성 
    - **결과**: **정답**
    - **이유**: 
        - 사용자가 SSH로 OS에 접속해 특정 패치 설치 또는 설정 변경 가능 
        - 다른 리전에 읽기 전용 복제본을 생성해 메인 리전 장애 시 복제본 승격시켜 즉시 서비스 가능 
- [ ] **D**: Oracle용 RDS로 마이그레이션 + 다른 가용영역에 대기 DB 생성 
    - **이유**: 
        - 고가용성엔 최적
        - OS 액세스 불가 및 Multi-AZ는 재해 복구보다는 장애 조치용
