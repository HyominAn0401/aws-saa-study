### 📝 667. 읽기 부하 분산 
Aurora를 사용하는 회사에서 대규모 보고서를 생성할 때 애플리케이션 성능이 느려지고 ReadIOPS 및 CPUUtilization 지표가 급증 

---

### 🎯 1. 요구사항 분석
* **ReadIOPS, CPUUtilization 급증**: 보고서를 만들기 위해 데이터를 '읽는' 과정에서 부하가 발생해 DB까지 느려지는 상황
    - 읽기 전용 복제본을 만들고 애플리케이션은 메인 DB를 사용하고, 보고서 생성 쿼리만 복제본 엔드포인트로 보내기 

---

### 🔍 2. 핵심 서비스 분석 

| 서비스/용어명 | 핵심 특징 | 비고 |
| :--- | :--- | :--- |
| **Amazon Redshift** | 완전 관리형 데이터 웨어하우스<br>방대한 양의 데이터를 모아두고 아주 빠른 속도로 분석해 비즈니스 인사이트를 얻게 해주는 거대한 데이터 창고 | 데이터 이전 비용 및 추가 리소스 비용 발생 | 

---

### ✅ 3. 선택지 분석 및 풀이

- [ ] **A**: Amazon Redshift
    - **이유**: Redshift는 분석에 최적화되어 있으나 월별 보고서를 위해 전체 데이터 파이프라인을 구축하는 것은 비용 비효율적 
- [X] **B**: 읽기 복제본 
    - **결과**: **정답**
    - **이유**: 가장 적은 비용으로 읽기 부하 분리 가능. 보고서용 엔드포인트만 복제본으로 변경하면 운영DB의 CPU/IOPS 부하 즉시 제거 가능 
        - 데이터를 복제한다는 의미 X 
- [ ] **C**: 더 큰 인스턴스 클래스로 마이그레이션
    - **이유**: 전체 비용이 크게 증가 
- [ ] **D**: 프로비저닝된 IOPS 늘림
    - **이유**: I/O 병목은 완화, 보고서 쿼리로 인한 CPU 점유율 문제는 해결되지 않으며 비용이 비쌈 

---

### 💡 핵심 요약
1. 기존: 애플리케이션&보고서 -> 메인 DB (Writer)
2. 변경: 애플리케이션 -> 메인 DB (Writer) / 보고서 -> 복제본 (Reader)
